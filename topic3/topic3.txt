## 1. 발표 목적과 청중

- 발표 제목: "Google Antigravity IDE: 에이전트 중심 개발의 실제"
- 부제: "핵심 기능, 실무 사용법, VS Code와의 차이"
- 대상: AI 코딩 도구에 관심 있는 초급~중급 개발자와 기술 리더
- 발표 시간: 약 20분
- 슬라이드 수: 정확히 13장
- 핵심 목표:
  1. Antigravity IDE가 무엇인지 이해시킨다.
  2. 설치 후 첫 작업을 완료하는 흐름을 구체적으로 보여준다.
  3. VS Code에도 강력한 에이전트 기능이 있다는 최신 상황을 반영해 공정하게 비교한다.
  4. 어떤 상황에서 어느 도구를 선택할지 판단 기준을 제공한다.
- 정보 기준일: 2026년 6월 21일. 이후 변경될 수 있는 버전, 요금, 지원 모델, 사용량 제한은 단정하지 말고 "발표 시점 공식 페이지 확인 필요"라고 표시한다.

## 2. 최종 산출물

- 브라우저에서 바로 실행 가능한 단일 HTML 파일 `topic3.html`을 작성한다.
- Reveal.js 같은 외부 프레젠테이션 라이브러리는 사용하지 않는다. HTML, CSS, JavaScript만 사용한다.
- 16:9 화면에 맞춘 13개의 전체 화면 슬라이드를 구현한다.
- 좌우 방향키, Page Up/Down, Space 키, 화면 버튼으로 이동할 수 있게 한다.
- 현재 슬라이드 번호와 전체 슬라이드 수를 하단에 표시한다.
- 인쇄 또는 PDF 저장 시 슬라이드별로 페이지가 나뉘도록 `@media print`를 작성한다.
- 모든 슬라이드에 짧은 발표자 노트를 HTML 주석 또는 별도 `<aside class="speaker-notes">` 요소로 포함한다.
- 최종 응답에는 설명을 덧붙이지 말고 완성된 HTML 전체 코드만 출력한다.

## 3. 시각 디자인

- 분위기: 미래지향적이지만 과장되지 않은 개발자 도구 프레젠테이션
- 배경: 짙은 네이비 또는 차콜 계열
- 강조색: Antigravity는 Google 계열의 블루·보라 그라데이션, VS Code는 익숙한 블루 계열

- 공식 로고와 스크린샷을 사용할 때는 비율을 변경하거나 임의로 재디자인하지 않는다.
- 각 슬라이드 하단에 해당 내용의 공식 출처를 10~12px 크기로 간결하게 표시한다.

## 4. 슬라이드별 구성

### 슬라이드 1. 표지 — 코드를 돕는 AI에서, 일을 맡는 에이전트로

- 제목: "Google Antigravity IDE"
- 부제: "에이전트 중심 개발의 특징·사용법·VS Code와의 차이"
- 핵심 문장: "개발자가 모든 코드를 직접 입력하는 흐름에서, 목표를 정의하고 에이전트의 작업을 검토하는 흐름으로"
- 시각자료: Editor, Terminal, Browser가 Agent Manager를 중심으로 연결된 단순 개념도
- 발표자 노트: Antigravity를 단순 자동완성 도구가 아니라 에이전트 중심 개발 환경으로 소개한다. 제품의 우열보다 작업 방식의 차이를 살펴보는 발표임을 밝힌다.

### 슬라이드 2. Antigravity IDE란 무엇인가

- Google이 제공하는 에이전트 중심 개발 플랫폼의 IDE 제품임을 설명한다.
- IDE 안에 코드 편집기뿐 아니라 Agent Manager, Artifacts, 코드베이스 이해 기능이 통합되어 있음을 보여준다.
- 에이전트가 편집기, 터미널, 브라우저를 사용해 계획·구현·검증까지 수행하는 구조를 설명한다.
- Antigravity 플랫폼에는 IDE 외에도 CLI와 SDK가 존재하지만, 이번 발표 범위는 IDE임을 명확히 한다.
- 시각자료: "IDE = Editor + Agent Manager + Artifacts + Browser verification" 4요소 다이어그램
- 발표자 노트: Antigravity를 클라우드 원격 IDE라고 단정하지 않는다. 현재 공식 설명에 맞춰 설치형 에이전트 IDE로 다룬다.

### 슬라이드 3. 핵심 차별점 ① Agent Manager와 병렬 작업

- Agent Manager를 여러 작업 공간과 에이전트를 관리하는 관제 화면으로 설명한다.
- 서로 독립적인 작업은 여러 에이전트에 분리해 병렬로 맡길 수 있음을 보여준다.
- 비동기 작업 상태를 확인하고 필요한 순간에 Editor View로 전환하는 Manager-to-Editor handoff를 설명한다.
- "병렬 실행 = 항상 더 빠름"이라고 주장하지 말고, 작업 간 의존성이 낮을 때 효과적이라고 설명한다.
- 시각자료: 한 명의 개발자가 기능 구현·테스트·문서화 에이전트를 감독하는 작업 보드
- 발표자 노트: 에이전트를 많이 실행하는 것이 목적이 아니라 작업을 잘 분해하고 검토 지점을 두는 것이 핵심이라고 강조한다.

### 슬라이드 4. 핵심 차별점 ② Artifacts로 작업 과정을 검토

- 에이전트가 단순 채팅 답변이 아니라 계획, 작업 목록, 구현 결과, 검증 증거 같은 Artifacts로 진행 상황을 전달한다고 설명한다.
- 사용자는 결과만 받는 대신 중간 계획과 증거를 검토하고 피드백할 수 있다.
- Artifacts가 AI의 정확성을 보장하는 것은 아니며, 사람이 판단할 수 있는 관찰 가능성을 높이는 장치임을 명시한다.
- 시각자료: Plan → Implementation → Test result → Screenshot/Video로 이어지는 증거 카드
- 발표자 노트: "에이전트가 했으니 믿는다"가 아니라 "에이전트가 남긴 근거를 보고 승인한다"는 방식으로 설명한다.

### 슬라이드 5. 핵심 차별점 ③ 브라우저 검증과 시각적 피드백

- 에이전트가 Chrome에서 페이지를 탐색하고 상호작용하며 자신의 코드를 브라우저 안에서 검증할 수 있음을 설명한다.
- 스크린샷과 영상 같은 Visual Artifacts로 결과를 확인할 수 있음을 보여준다.
- 화면의 특정 위치에 시각적 코멘트를 남겨 수정 범위를 구체적으로 전달하는 흐름을 설명한다.
- 프론트엔드의 구현→확인→피드백 반복에서 컨텍스트 전환을 줄이는 사례를 제시한다.
- 시각자료: Code change → Browser test → Screenshot → Visual comment → Revision 순환 다이어그램
- 발표자 노트: 브라우저 자동화 결과도 테스트 코드, 접근성 검사, 실제 사용자 검토를 대체하지 않는다고 덧붙인다.

### 슬라이드 6. 시작하기 — 설치와 초기 설정

- 공식 다운로드 페이지에서 macOS, Windows, Linux용 설치 파일을 선택한다.
- 설치 후 Google 계정 또는 조직이 제공한 인증 방식으로 로그인한다.
- 프로젝트 폴더 또는 저장소를 열고, 필요한 설정·키 바인딩·확장 기능의 가져오기 가능 여부를 온보딩 화면에서 확인한다.
- Agent Permissions, Secure Mode, Sandbox 관련 설정을 확인하고 자동 실행 범위를 프로젝트 위험도에 맞게 조정한다.
- 온보딩 UI와 지원 방식은 버전에 따라 달라질 수 있으므로 화면의 최신 안내를 따른다고 표시한다.
- 시각자료: Download → Sign in → Open workspace → Permissions의 4단계 타임라인
- 발표자 노트: 회사 코드에서는 개인 계정 사용 여부, 데이터 정책, 조직 관리 정책을 먼저 확인해야 한다.

### 슬라이드 7. 첫 작업 — 좋은 요청을 만드는 공식

- 프롬프트 구성: 목표 + 관련 범위 + 제약조건 + 완료 기준 + 검증 방법
- 나쁜 예: "로그인 기능 만들어줘"
- 좋은 예: "`src/auth` 범위에서 이메일 로그인 오류를 수정하라. 공개 API는 유지하고, 실패 테스트를 먼저 재현한 뒤 관련 테스트와 린트를 통과시켜라. 변경 파일과 검증 결과를 Artifact로 요약하라."
- 큰 작업은 계획 수립과 구현을 분리하고, 먼저 계획 Artifact를 검토한 뒤 실행한다.
- 시각자료: 모호한 한 줄 요청과 구조화된 요청을 좌우 비교
- 발표자 노트: 에이전트의 품질은 모델뿐 아니라 문제 정의와 완료 기준의 품질에 크게 좌우된다고 설명한다.

### 슬라이드 8. 실무 워크플로 — 요청부터 승인까지

- 1단계: Agent Manager에서 작업과 완료 기준을 전달한다.
- 2단계: 에이전트가 코드베이스를 조사하고 계획 Artifact를 만든다.
- 3단계: 사용자가 계획, 권한 요청, 변경 범위를 검토한다.
- 4단계: 에이전트가 파일 수정, 명령 실행, 테스트 또는 브라우저 검증을 수행한다.
- 5단계: diff와 Artifacts를 검토하고 구체적인 피드백을 남긴다.
- 6단계: 최종 테스트와 Git 상태를 사람이 확인한 뒤 커밋 또는 PR 단계로 넘긴다.
- 시각자료: Request → Plan → Approve → Execute → Verify → Commit의 가로 플로차트
- 발표자 노트: 삭제, 배포, 자격 증명 접근, 외부 전송 같은 고위험 작업에는 명시적인 승인 단계를 둔다.

### 슬라이드 9. 병렬 에이전트를 안전하게 쓰는 방법

- 병렬화하기 좋은 작업: 서로 다른 모듈 조사, 독립 테스트 작성, 문서 초안, 대안 비교
- 순차 실행이 필요한 작업: 공통 인터페이스 변경 후 소비 코드 수정, DB 마이그레이션 후 배포, 같은 파일을 동시에 수정하는 작업
- 각 에이전트에 파일 범위, 산출물, 금지 사항, 완료 기준을 명확히 지정한다.
- 작업 간 충돌을 막기 위해 작은 범위, 별도 브랜치 또는 작업 공간, 통합 검토 단계를 사용한다.
- 시각자료: "병렬화 적합"과 "순차 실행 필요"를 나누는 2열 카드
- 발표자 노트: 속도를 높이기 전에 충돌 비용과 리뷰 부담을 계산해야 한다.

### 슬라이드 10. Antigravity와 VS Code — 공통점부터 정확히 보기

- 두 도구 모두 자연어 요청을 기반으로 계획, 여러 파일 편집, 명령 실행, 테스트와 자기 수정이 가능한 에이전트형 개발을 지원할 수 있다.
- 최신 VS Code에는 Agents window와 Chat view가 있으며, 로컬 에이전트뿐 아니라 Copilot CLI, 클라우드 및 서드파티 에이전트 세션을 다룬다.
- 두 도구 모두 사람의 검토, 권한 제어, 컨텍스트 관리가 필요하다.
- 따라서 "Antigravity는 에이전트이고 VS Code는 자동완성 도구"라는 낡은 비교를 사용하지 않는다.
- 시각자료: 겹치는 영역에 Agentic editing, Terminal, Planning, Testing, Browser를 배치한 벤다이어그램
- 발표자 노트: VS Code의 기능은 빠르게 변하므로 차별점은 기능 체크리스트 한 줄보다 기본 UX와 생태계에서 찾아야 한다.

### 슬라이드 11. 차이는 어디에 있는가 — 기본 철학과 생태계

- 표의 열: 비교 항목 / Antigravity IDE / VS Code
- 비교 행:
  - 기본 철학: 에이전트 관제와 검증 중심 / 범용 편집기에 에이전트 기능 통합
  - 대표 작업 화면: Agent Manager + Editor View / Agents window + Chat view + Editor
  - 검증 경험: Artifacts, 브라우저 사용, 시각적 피드백이 핵심 UX / 에이전트·통합 브라우저·확장 기능을 조합하는 폭넓은 방식
  - AI 생태계: Google·Gemini와 긴밀하게 최적화 / GitHub Copilot 중심이면서 모델·에이전트·확장 선택 폭이 큼
  - 확장성과 도구 성숙도: 에이전트 워크플로에 집중된 신생 플랫폼 / 방대한 확장 마켓과 오랜 언어·디버깅 생태계
  - 병렬 작업: Agent Manager를 전면에 둔 멀티에이전트 관제 / Agents window에서 다양한 에이전트 세션 관리
- 어느 한쪽을 일방적으로 우월하다고 결론 내리지 않는다.
- 발표자 노트: VS Code의 정확한 기능 명칭과 범위는 공식 문서 기준으로 표기한다.

### 슬라이드 12. 어떤 도구를 선택할 것인가

- Antigravity가 잘 맞는 경우:
  - 에이전트 병렬 작업과 Artifact 기반 리뷰를 기본 흐름으로 사용하고 싶다.
  - 웹·풀스택 작업에서 브라우저 검증과 시각 피드백을 자주 반복한다.
  - Google/Gemini 중심 환경과의 결합이 중요하다.
- VS Code가 잘 맞는 경우:
  - 기존 확장, 디버거, 언어 도구, Dev Container, 조직 표준을 유지해야 한다.
  - 여러 모델·에이전트·확장을 선택하고 세밀하게 조합하고 싶다.
  - 팀의 기존 학습 비용과 운영 안정성이 더 중요하다.
- 결론: 작은 실제 과제를 두 도구에서 동일한 완료 기준으로 수행해 품질, 시간, 검토 부담을 측정한 뒤 선택한다.
- 시각자료: Yes/No 선택 트리 또는 가중치 평가표
- 발표자 노트: 개인 취향이 아니라 저장소 적합성, 보안 정책, 리뷰 비용, 팀 표준으로 평가하라고 제안한다.

### 슬라이드 13. 결론 — 에이전트를 쓰는 능력은 검토하는 능력

- 핵심 요약 1: Antigravity의 중심은 Agent Manager, Artifacts, 브라우저 검증이 결합된 에이전트 중심 워크플로다.
- 핵심 요약 2: VS Code 역시 강력한 에이전트 플랫폼이므로 차이는 "가능/불가능"보다 기본 경험과 생태계에 있다.
- 핵심 요약 3: 좋은 결과를 위해서는 명확한 완료 기준, 최소 권한, diff·테스트·Artifact 검토가 필요하다.
- 마무리 문장: "코드 작성 속도보다, 작업을 분해하고 증거를 검토하는 능력이 더 중요해진다."
- 시각자료: Intent → Delegation → Evidence → Review의 4단계 루프
- 발표자 노트: Q&A를 유도하고, 공식 문서에서 최신 버전과 정책을 다시 확인하도록 안내한다.

## 5. 사실 검증 및 표현 규칙

- 아래 공식 출처를 우선 사용하고, 각 주장과 가장 가까운 슬라이드 하단에 URL을 표시한다.
- 공식 출처에서 확인하지 못한 기능, 시장점유율, 생산성 향상률, 속도 비교 수치는 넣지 않는다.
- "완전 자율", "오류 없음", "VS Code에는 없는 기능", "무조건 더 빠름" 같은 표현을 금지한다.
- Artifacts는 신뢰를 높이는 검토 수단으로 설명하며 정확성 보장 장치로 표현하지 않는다.
- 제품 버전과 요금제는 별도의 핵심 비교 기준으로 단정하지 않는다. 필요하면 기준일과 변동 가능성을 함께 쓴다.
- VS Code의 에이전트 기능을 과거의 Copilot 자동완성 수준으로 축소하지 않는다.
- Antigravity가 로컬 자원을 전혀 사용하지 않는 순수 클라우드 IDE라는 식의 확인되지 않은 아키텍처 설명을 넣지 않는다.
- 설치 및 사용 화면은 최신 버전에 따라 다를 수 있다는 주석을 넣는다.
