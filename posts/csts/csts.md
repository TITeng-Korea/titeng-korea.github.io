# 1. 요구 사항 개요

1. **요구사항이란**
    1. 어떤 문제를 해결하거나 특정의 목적을 위하여 필요한 조건이나 능력
    2. 소프트웨어가 수행해야 하는 작업 및 성공을 위해 충족해야 하는 조건
2. **요구사항분류**
    1. **기능 요구사항**
    반드시 수행해야 하는 기능(동작)
    2. **성능 요구사항**
    목표 시스템의 성능
    3. **시스템 장비구성 요구사항**
    필요한 시스템 장비 구성
    4. **인터페이스 요구사항**
    시스템 인터페이스, 사용자 인터페이스
    5. **데이터 요구사항**
    데이터 구축
    6. **테스트 요구사항**
    테스트 유형, 테스트 환경, 방법, 절차
    7. **보안 요구사항**
    접근 통제
    8. **품질 요구사항**
    신뢰성, 사용성, 유지보수성, 이식성, 보안성
    9. **제약사항**
    목표시스템 설계, 구축, 운영과 관련된 제도 등 제약 조건
    10. **프로젝트 관리 요구사항**
    관리 방법, 추진 단계별 수행방안
    11. **프로젝트 지원 요구사항**
    시스템 안정화 및 운영, 훈련 및 지원, 하자보수, 유지관리

# 2. 테스트 개요

## SW 테스팅 7가지 기본 원칙

1. **테스팅은 결함이 존재함을 밝히는 활동이다.**
2. **완벽한 테스팅은 불가능하다.**
무한 경로, 무한 입력값, 무한 타이밍 등 모든 조합을 전수 테스트 불가능
3. **테스팅은 개발초기에 시작해야한다.**
비용/품질/기간 준수 가능
4. **결함은 집중된다.** 
8:2 파로스 원칙: 대다수의 결함은 소수의 특정 모듈에 집중발생된다.
5. **살충제 패러독스**
동일한 테스트 케이스의 반복으로는 새로운 결함을 찾을 수 없다.
6. **테스팅은 정황에 의존적이다.**
정황과 도메인에 따라 다르게 테스트해야 한다.
7. **오류 부재의 궤변**
요구사항과 불일치하면 결함이 없어도 품질이 높다고 할 수 없다.

## G.J 마이어스 SW테스트 원칙

1. 테스트는 프로그램을 개발한 팀과 무관한 그룹이 수행해야 한다.
2. 결함이 발견되지 않으리라는 가정 하에 테스트 계획을 수립해선 안된다.
3. 타당한 경우 뿐만 아니라 타당하지 않고 예상하지 못한 경우에 대해 테스트해야 한다.
4. 프로그램의 어떤 부분에 남아 있을 확률은 그 부분에서 이미 발견된 결함의 수에 비례한다.
5. 테스트 케이스를 체계적으로 관리하라.
6. 테스트 결과를 철저히 점검하라.

# 3. 테스트 방법

## 테스트 레벨에 의한 방법

| 테스트 기법 | 설명 |
| --- | --- |
| 컴포넌트(Component)/
단위(Unit) 테스트 |   • 테스트가 가능한 단위로 나누어진 소프트웨어 내에서 결함을 찾고 그 기능을 검증
  • 주로 구조 기반 테스트
  • 시스템의 다른 부분에서 격리해 독립적으로 수행
  • 소스 코드를 활용하여 테스트를 설계, 기능적 테스트 수행 |
| 통합(Intergration) 테스트 |   • 시스템을 구성하는 단위 모듈들이 정확하게 통합되었는지 초점을 둠
  • 시나리오 기반의 테스트 수행
  • 모듈 간 통합에 주목하여, 코드가 외부 종속성과 올바르게 작동하는지 확인 |
| 시스템(System) 테스트 |   • 발견할 수 있는 환경특성 장애 리스크를 최소화하기 위해 실제 최종사용환경 또는 유사한 환경에서 수행
  • 기능적 요구사항은 명세기반(Black-box) 기법 수행
  • 비기능적 요구사항은 신뢰성, 사용성, 효율성, 유지보수성, 이식성, 보안성, 호환성 등 품질 속성을 포함
  • End to End 테스트를 함(종단간 테스트) |
| 인수(Acceptance) 테스트 |   • 고객/사용자 관점에서 고객이 기대하는 방식으로 소프트웨어가 동작하는 지 확인 |

## 정적 테스트

### 유형

1. 코딩 표준: MISRA-C 등의 표준 코딩에 대한 준수 여부 검사
2. 복잡도 측정: 사이클로매틱 복잡도 등, 프로그램의 복잡도 측정
3. 자료 흐름 분석: 프로그램 자료 흐름에 이상 존재 여부 분석

### 리뷰(Review)

소프트웨어의 다양한 산출물에 존재하는 결함을 검출 또는 진행상황을 점검하기 위한 활동

| 리뷰 유형 | 설명 |
| --- | --- |
| 관리 리뷰 |   • 프로젝트 진행 상황에 대한 검토를 바탕으로 일정, 인력, 범위 등에 대한 통제 및 의사결정 지원 |
| 기술 리뷰 |   • 정의된 계획 및 명세를 준수하고 있는지에 대한 검토 수행 |
| 인스펙션 |   • 문제(Anomaly)를 식별하고 문제에 대한 올바른 해결(Resolution)을 검증 |
| 워크쓰루 |   • 문제를 식별하고 더 나아가서 대안조사, 개선활동, 학습 기회 제공을 수행 |
| 감사 |   • 객관적인 표준과 규제에 대한 준수를 독립적으로 평가 |

## 동적 테스트

### 경험 기반 테스트(Heuristic 기반 테스트)

**지식이나 경험으로 테스트 케이스를 도출하는 방법**

1. **탐색적 테스트**
    - 테스트 차터(Test Charter)
    - Time-Boxing
2. **오류 추정**
    - 가능한 결함 나열
    - 이런 유형의 결함을 공격할 수 있는 테스트 설계
3. **체크리스트 테스트**
    - 테스트 및 평가할 내용 나열
    - 체크리스트 기반으로 테스트

---

### 명세 기반 테스트(Black-Box)

1. **동등 분할 기법**
    - 대표 값을 이용한 테스트 케이스 도출 기법
    - 특정 입력 값이 같은 결과 값을 출력할 경우, 이러한 범위를 하나의 클래스로 간주
    - 각 클래스의 원소 중 대푯값을 하나 선택하여 테스트 케이스 작성
    - 유효하지 않은 데이터도 포함 가능
    - 모든 테스트 레벨에서 적용 가능
    
2. **경계 값 분석**
    - 동등 분할의 경계 부분에 해당하는 입력 값을 테스트 케이스로 작성
    - 최대 값과 최소 값은 그 영역의 경계 값이 됨
    - 결함은 경계 값 근처에서 많이 발생한다는 점을 이용
    - 경계 값 분석은 모든 테스트 레벨에서 적용 가능
    
3. **결정 테이블**
    - 입력 조건에 따른 기대 결과의 참/거짓을 테스트 케이스로 도출
    - 논리적인 조건이나 상황을 구현하는 시스템에서 요구 사항 도출 또는 내부 시스템 설계를 문서화하는데 유용
    - 결정 테이블의 각 컬럼당 하나 이상의 테스트 케이스를 생성
        - 장점: 논리적으로 의존적인 가능한 모든 조건들의 조합을 생성
        - 단점: 작성에 많은 시간 소요, 복잡한 시스템 표현 어려움, 실수 가능성
    - **결정 테이블 실습**

### 구조 기반 테스트

- 커버리지: 시스템 또는 소프트웨어의 구조가 테스트 묶음에 의해 테스트된 정보
- 커버리지 유형

| 구분 | 설명 |
| --- | --- |
| 구문 커버리지 | 전체 구문 중 실행된 구문이 몇 퍼센트인지 측정 |
| 결정 커버리지 | 전체 조건식이 참, 거짓 각각 최소 한번씩 실행되면 충족 |
| 조건 커버리지 | 개별 조건식을 기준으로 참, 거짓 각각 한번씩 실행되면 충족 |
| 조건/결정 커버리지 | 전체 조건식과 개별 조건식 모두 각각 참, 거짓을 한번씩 실행되면 충족 |
| 변경 조건/결정 커버리지
(MC/DC) | 각 개별 조건식이 다른 개별 조건식과 무관하게 전체 조건식의 결과에 독립적으로 영향을 주도록 실행되면 충족
(결정 포인트 = 0, A = 0, B = 0 조합은 A, B가 각각 독립적으로 1로 변경되어도 결정 포인트에는 영향이 없음으로 제외) |
| 다중 조건 커버리지 | 모든 개별 조건식의 모든 가능한 논리적인 조합을 고려하여 100% 커버리지 보장 |

## 회귀(Regression) 테스트

- 유지 보수 단계에서 소프트웨어가 수정된 후에 변경이 올바르게 되었는지 검사하기 위한 테스트

| 테스트 방식 | 설명 |
| --- | --- |
| Retest-All |   • 기존에 개발된 모든 테스트 케이스를 사용하는 방식 |
| 선택적 리그레션 테스트 |   • 기존의 테스트 케이스 중에서 일부만 선정
  • 전후 서로 다른 경과를 출력할 가능성이 있는 테스트 케이스를 식별하여 수행 |
| 테스트 최소화 |   • 중복된 테스트 케이스를 제거하여 테스트 케이스의 수를 줄이는 방식 |
| 테스트 우선 순위화 |   • 테스트 케이스에 우선 순위를 두어 우선 순위가 높은 테스트 케이스를 활용하는 방식 |

## 비기능 테스트

| 테스트 유형 | 설명 |
| --- | --- |
| 사용성 테스트(Usability Test) |   • 소프트웨어를 쉽게 사용할 수 있는 정도, 편의 기능 제공 정도를 측정하고 확인하는 테스트 |
| 성능 테스트(Performance Test) |   • 시스템에 부하를 주면서 응답 시간, 처리량, 속도 등 성능 지표 추이 측정 |
| 회복 테스트(Recovery Test) |   • 문제가 발생했을 때, 복귀 능력 검증 |
| 보안 테스트(Security Test) |   • 외부의 침입이나 해킹에 대응하는 능력 검증 |

## 성능 테스트

- 유지 보수 단계에서 소프트웨어가 수정된 후에 변경이 올바르게 되었는지 검사하기 위한 테스트

| 종류 | 설명 |
| --- | --- |
| 부하 테스트(Load Test) |   • 사전에 결정된 Peak 시점의 부하 상황에서 시스템의 성능을 검증
(적정 부하 유지 여부 판별) |
| 스트레스 테스트(Stress Test) |   • 시스템 처리 능력 이상의 부하, 즉 임계점 이상의 부하를 가하여 비정상적인 상황에서 처리를 테스트
(적정 부하 2배 유지) |
| 내구성 테스트(Endurance Test) |   • 오랜 시간 동안 시스템에 높은 부하를 가하여 시스템의 반응을 파악. 메모리 누수 감지 등
(8시간 이상의 장시간 부하) |
| 스파이크 테스트(Spike Test) |   • 갑작스런 사용량 급증에 대한 테스트
(순간적인 대량 부하) |
| 시스템 중단점 테스트(Breakpoint Test) |   • 동시 단말 사용자의 수를 점진적으로 증가시켜, 시스템의 장애 지점을 결정 |

## 테스트 프로세스 개요

- **성공적인 테스트를 위해 효과적인 테스트 접근 방법뿐만 아니라 체계적인 테스트 수행을 위한 계획 수립과 관리 필요**
- **테스트 프로세스 구성**
    1. **조직 테스트 프로세스**
        1. 조직 테스트 명세 개발
        2. 조직 테스트 명세 활용 모니터링 및 제어
        3. 조직 테스트 명세 갱신
    2. **테스트 관리 프로세스**
        1. 테스트 계획
        2. 테스트 모니터링 및 제어
        3. 테스트 종료
    3. **동적 테스트 프로세스**
        1. 테스트 설계 및 구현
        2. 테스트 환경 구축 및 관리
        3. 테스트 실행
        4. 결함 보고

## 조직 테스트 프로세스

- **조직 전체에서 수행될 모든 테스트 프로젝트의 공통적인 사항을 정의하는 데 초점**

| 항목 | 설명 |
| --- | --- |
| 조직 테스트 명세 개발 |   • 조직의 테스트 목표를 바탕으로 조직 테스트 정책 명세서 및 조직 테스트 전략 명세 개발 |
| 조직 테스트 명세 활용 모니터링 및 제어 |   • 조직 테스트 명세서가 조직 내에서 효과적으로 활용되는 지를 모니터링하고 필요하면 테스트 관리 활동을 제어 |
| 조직 테스트 명세 갱신 |   • 조직 테스트 명세서의 활용에 대한 피드백을 바탕으로 조직 테스트 명세서를 갱신 |

## 조직 테스트 정책 명세

| 항목 | 설명 |
| --- | --- |
| 테스트 목적 |   • 조직에서의 테스트에 대한 목적, 목표, 테스트 범위 기술 |
| 테스트 프로세스 |   • 조직이 테스트를 수행할 때 준수할 테스트 프로세스 명시 |
| 테스트 조직 및 역할 |   • 테스트 활동을 수행하는 테스트 조직의 구조와 역할 정의 |
| 테스트 표준 |   • 테스트를 수행하면서 참고하고 준수해야 할 표준 정의 |
| 테스트 자산 관리 |   • 테스트 결과물을 축적하고 재사용하는 방법 정의 |
| 테스트 프로세스 개선 |   • 테스트 프로세스를 평가하고 개선하는 방법 정의 |

## 조직 테스트 전략 명세

| 항목 | 설명 |
| --- | --- |
| 위험 관리 |   • 테스트를 진행하면서 수행할 위험 관리 방법을 명시 |
| 테스트 선택 및 우선 순위 |   • 우선순위를 부여하고, 테스트를 선택하는 방법을 명시 |
| 테스트 문서화 |   • 테스트를 수행하면서 작성할 산출물을 명시 |
| 형상 관리 |   • 테스트 산출물에 대한 형상 관리 방법을 명시 |
| 결함 관리 |   • 검출된 결함을 종결시킬때까지의 관리 방법을 명시 |
| 자동화 도구 |   • 테스트 활동을 자동화하기 위한 도구들을 명시 |
| 수행 개별 테스트 |   • 레벨 테스트 및 유형 테스트 등 수행할 개발 테스트 명시 |

## 동적 테스트 프로세스

1. **테스트 설계 및 구현**
    - 테스트 실행 프로세스에서 실행될 테스트 절차를 도출
    - 테스트 basis를 분석하여 테스트 요구 사항/조건/커버리지 기준 및 테스트 케이스 등을 도출
2. **테스트 환경 구축 및 관리**
    - 테스트를 수행하는 데 필요한 테스트 환경 요건과 테스트 데이터 요건을 바탕으로 테스트 환경을 구축하고 테스트 데이터를 준비하여 테스트 실행이 진행되도록 지원
3. **테스트 실행**
    - 테스트 설계 및 구현 활동에서 개발된 테스트 절차를 실행함으로써 수행
4. **결함 보고**
    - 테스트 결과를 분석하여 식별한 결함을 관계자들에게 보고하고 결함 보고서를 업데이트
