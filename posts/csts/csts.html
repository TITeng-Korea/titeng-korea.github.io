1. 요구사항 개요 03 / 49

요구사항이란

정의 1

어떤 문제를 해결하거나 특정의 목적을 위하여 필요한 조건이나 능력

정의 2

소프트웨어가 수행해야 하는 작업 및 성공을 위해 충족해야 하는 조건
테스트는 결국 요구사항을 기준으로 합니다. 요구사항이 불명확하면 테스트도 모호해지고, 반대로 요구사항이 선명하면 테스트 설계 기준도 선명해집니다.
1. 요구사항 개요 04 / 49

요구사항 분류

  • 기능 요구사항: 반드시 수행해야 하는 기능(동작)
  • 성능 요구사항: 목표 시스템의 성능
  • 시스템 장비구성 요구사항: 필요한 시스템 장비 구성
  • 인터페이스 요구사항: 시스템 인터페이스, 사용자 인터페이스
  • 데이터 요구사항: 데이터 구축
  • 테스트 요구사항: 테스트 유형, 테스트 환경, 방법, 절차
  • 보안 요구사항: 접근 통제
  • 품질 요구사항: 신뢰성, 사용성, 유지보수성, 이식성, 보안성
  • 제약사항: 목표시스템 설계, 구축, 운영과 관련된 제도 등 제약 조건
  • 프로젝트 관리 요구사항: 관리 방법, 추진 단계별 수행방안
  • 프로젝트 지원 요구사항: 시스템 안정화 및 운영, 훈련 및 지원, 하자보수, 유지관리
요구사항은 기능만 의미하지 않습니다. 성능, 품질, 운영, 지원까지 모두 테스트 기준으로 연결됩니다.
2. 테스트 개요 06 / 49

테스트 개요

이번 장의 축

  • SW 테스팅 7가지 기본 원칙
  • G.J 마이어스 SW테스트 원칙
  • 실무에서 테스트를 설계할 때의 사고 기준

왜 중요한가

  • 테스트 기법을 암기해도 원칙을 모르고 적용하면 기계적인 체크리스트가 되기 쉽습니다.
  • 원칙은 “무엇을 테스트할지”보다 “어떻게 생각해야 하는지”를 정리해 줍니다.
2. 테스트 개요 07 / 49

SW 테스팅 7가지 기본 원칙

  1. 테스팅은 결함이 존재함을 밝히는 활동이다.
  2. 완벽한 테스팅은 불가능하다. 무한 경로, 무한 입력값, 무한 타이밍 등 모든 조합을 전수 테스트 불가능
  3. 테스팅은 개발초기에 시작해야한다. 비용/품질/기간 준수 가능
  4. 결함은 집중된다. 8:2 파로스 원칙: 대다수의 결함은 소수의 특정 모듈에 집중발생된다.
  5. 살충제 패러독스 동일한 테스트 케이스의 반복으로는 새로운 결함을 찾을 수 없다.
  6. 테스팅은 정황에 의존적이다. 정황과 도메인에 따라 다르게 테스트해야 한다.
  7. 오류 부재의 궤변 요구사항과 불일치하면 결함이 없어도 품질이 높다고 할 수 없다.
즉, 테스트는 개수를 많이 돌리는 것이 아니라 상황과 목적에 맞게 설계해야 의미가 있습니다.
2. 테스트 개요 09 / 49

G.J 마이어스 SW테스트 원칙

  1. 테스트는 프로그램을 개발한 팀과 무관한 그룹이 수행해야 한다.
  2. 결함이 발견되지 않으리라는 가정 하에 테스트 계획을 수립해선 안된다.
  3. 타당한 경우 뿐만 아니라 타당하지 않고 예상하지 못한 경우에 대해 테스트해야 한다.
  4. 프로그램의 어떤 부분에 남아 있을 확률은 그 부분에서 이미 발견된 결함의 수에 비례한다.
  5. 테스트 케이스를 체계적으로 관리하라.
  6. 테스트 결과를 철저히 점검하라.

3. 테스트 방법 11 / 49

테스트 레벨에 의한 방법

테스트 기법 설명
컴포넌트(Component)/단위(Unit) 테스트 • 테스트가 가능한 단위로 나누어진 소프트웨어 내에서 결함을 찾고 그 기능을 검증
• 주로 구조 기반 테스트
• 시스템의 다른 부분에서 격리해 독립적으로 수행
• 소스 코드를 활용하여 테스트를 설계, 기능적 테스트 수행
통합(Intergration) 테스트 • 시스템을 구성하는 단위 모듈들이 정확하게 통합되었는지 초점을 둠
• 시나리오 기반의 테스트 수행
• 모듈 간 통합에 주목하여, 코드가 외부 종속성과 올바르게 작동하는지 확인
시스템(System) 테스트 • 발견할 수 있는 환경특성 장애 리스크를 최소화하기 위해 실제 최종사용환경 또는 유사한 환경에서 수행
• 기능적 요구사항은 명세기반(Black-box) 기법 수행
• 비기능적 요구사항은 신뢰성, 사용성, 효율성, 유지보수성, 이식성, 보안성, 호환성 등 품질 속성을 포함
• End to End 테스트를 함(종단간 테스트)
인수(Acceptance) 테스트 • 고객/사용자 관점에서 고객이 기대하는 방식으로 소프트웨어가 동작하는 지 확인

• 일반적으로 시스템 테스트와 인수 테스트 수준에서 진행됨
3. 테스트 방법 14 / 49

테스트 설계기법에 의한 방법

테스트 설계기법 설명
정적 테스트 • 테스트 대상을 실행하지 않는 방식으로 테스트
동적 테스트 • 테스트 대상을 실행하여 결함을 검출하는 방식으로, 적절한 입력값 즉 테스트 케이스를 결정해야 함
정적 테스트는 실행 전 품질을 확인하는 방법이고, 동적 테스트는 실제 실행과 결과를 중심으로 판단하는 방법이에요.
3. 테스트 방법 15 / 49

정적 테스트

핵심 정의

정적 테스트는 테스트 대상을 실행하지 않는 방식으로 수행한다.

즉, 실행 결과보다 코드와 산출물 자체를 보고 이상을 찾는 접근이에요.

실무 의미

  • 구현 초기에 결함을 빠르게 찾을 수 있습니다.
  • 코드 실행 환경이 준비되기 전에도 수행 가능합니다.
  • 리뷰, 표준 준수 점검, 복잡도 분석과 연결됩니다.

3. 테스트 방법 17 / 49

리뷰(Review)

소프트웨어의 다양한 산출물에 존재하는 결함을 검출 또는 진행상황을 점검하기 위한 활동
리뷰 유형 설명
관리 리뷰• 프로젝트 진행 상황에 대한 검토를 바탕으로 일정, 인력, 범위 등에 대한 통제 및 의사결정 지원
기술 리뷰• 정의된 계획 및 명세를 준수하고 있는지에 대한 검토 수행
인스펙션• 문제(Anomaly)를 식별하고 문제에 대한 올바른 해결(Resolution)을 검증
워크쓰루• 문제를 식별하고 더 나아가서 대안조사, 개선활동, 학습 기회 제공을 수행
감사• 객관적인 표준과 규제에 대한 준수를 독립적으로 평가
3. 테스트 방법 18 / 49

동적 테스트

핵심 정의

동적 테스트는 테스트 대상을 실행하여 결함을 검출하는 방식이며, 적절한 입력값 즉 테스트 케이스를 결정해야 한다.

이 장에서 다룰 것

  • 경험 기반 테스트(Heuristic 기반 테스트)
  • 명세 기반 테스트(Black-Box)
  • 상태 전이 테스트
  • 구조 기반 테스트와 커버리지

경험 기반 테스트(Heuristic 기반 테스트)

3. 테스트 방법 20 / 49

명세 기반 테스트(Black-Box) 개요

  • 동등 분할 기법
  • 경계 값 분석
  • 결정 테이블
  • 상태 전이 테스트

핵심 관점

  • 내부 코드 구조보다 입력과 출력의 관계를 중심으로 봅니다.
  • 요구사항과 명세가 충실할수록 명세 기반 테스트 품질도 높아집니다.
3. 테스트 방법 21 / 49

동등 분할 기법

  • 대표 값을 이용한 테스트 케이스 도출 기법
  • 특정 입력 값이 같은 결과 값을 출력할 경우, 이러한 범위를 하나의 클래스로 간주
  • 각 클래스의 원소 중 대푯값을 하나 선택하여 테스트 케이스 작성
  • 유효하지 않은 데이터도 포함 가능
  • 모든 테스트 레벨에서 적용 가능
동등 분할의 목적은 모든 입력을 다 시험하는 대신, 같은 동작을 보일 가능성이 높은 입력군을 대표값으로 묶는 거예요.
3. 테스트 방법 22 / 49

경계 값 분석

  • 동등 분할의 경계 부분에 해당하는 입력 값을 테스트 케이스로 작성
  • 최대 값과 최소 값은 그 영역의 경계 값이 됨
  • 결함은 경계 값 근처에서 많이 발생한다는 점을 이용
  • 경계 값 분석은 모든 테스트 레벨에서 적용 가능
실무에서는 대표값만 보면 놓치는 경우가 많기 때문에, 경계 값 분석은 동등 분할과 거의 항상 같이 사용됩니다.
3. 테스트 방법 23 / 49

결정 테이블

  • 입력 조건에 따른 기대 결과의 참/거짓을 테스트 케이스로 도출
  • 논리적인 조건이나 상황을 구현하는 시스템에서 요구 사항 도출 또는 내부 시스템 설계를 문서화하는데 유용
  • 결정 테이블의 각 컬럼당 하나 이상의 테스트 케이스를 생성

장점과 단점

  • 장점: 논리적으로 의존적인 가능한 모든 조건들의 조합을 생성
  • 단점: 작성에 많은 시간 소요, 복잡한 시스템 표현 어려움, 실수 가능성

성능 테스트

유지 보수 단계에서 소프트웨어가 수정된 후에 변경이 올바르게 되었는지 검사하기 위한 테스트
종류 설명
부하 테스트(Load Test)• 사전에 결정된 Peak 시점의 부하 상황에서 시스템의 성능을 검증 (적정 부하 유지 여부 판별)
스트레스 테스트(Stress Test)• 시스템 처리 능력 이상의 부하, 즉 임계점 이상의 부하를 가하여 비정상적인 상황에서 처리를 테스트 (적정 부하 2배 유지)
내구성 테스트(Endurance Test)• 오랜 시간 동안 시스템에 높은 부하를 가하여 시스템의 반응을 파악. 메모리 누수 감지 등 (8시간 이상의 장시간 부하)
스파이크 테스트(Spike Test)• 갑작스런 사용량 급증에 대한 테스트 (순간적인 대량 부하)
시스템 중단점 테스트(Breakpoint Test)• 동시 단말 사용자의 수를 점진적으로 증가시켜, 시스템의 장애 지점을 결정

  • 테스트 설계 및 구현: 테스트 실행 프로세스에서 실행될 테스트 절차를 도출하고, 테스트 basis를 분석하여 테스트 요구 사항/조건/커버리지 기준 및 테스트 케이스 등을 도출
  • 테스트 환경 구축 및 관리: 테스트를 수행하는 데 필요한 테스트 환경 요건과 테스트 데이터 요건을 바탕으로 테스트 환경을 구축하고 테스트 데이터를 준비하여 테스트 실행이 진행되도록 지원
  • 테스트 실행: 테스트 설계 및 구현 활동에서 개발된 테스트 절차를 실행함으로써 수행
  • 결함 보고: 테스트 결과를 분석하여 식별한 결함을 관계자들에게 보고하고 결함 보고서를 업데이트

최종 정리

  • 요구사항은 테스트의 출발점이에요.
  • 원칙은 테스트 설계의 사고 기준이에요.
  • 기법은 입력, 상태, 구조를 서로 다른 관점에서 검증하게 해줍니다.
  • 프로세스는 테스트 활동을 조직 수준에서 반복 가능하게 만듭니다.
