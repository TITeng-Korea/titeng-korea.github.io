Git 역할 재정의

Git은 백업 도구 이상

1
변경의 기록
코드가 언제, 왜, 어떻게 바뀌었는지 남기는 사고의 흔적. 최종 결과보다 과정 확인
2
실험의 경계
새 기능, 라이브러리 변경, AI 리팩터링을 안전하게 시도하는 공간. 실패해도 돌아갈 지점 확보
3
협업의 언어
말로 설명하는 대신 diff와 PR로 변경 파일, 변경 줄, 변경 이유를 공유
메시지 Git의 진짜 가치는 “백업”보다 “변경 설명력”

AI 요청 범위

한 번에 너무 많이 맡기지 않기

나쁜 요청 전체 프로젝트 구조 개선하고, 버그도 고치고, UI도 예쁘게 바꾸고, 테스트도 추가해줘. 수정 범위 제한 없음 검토 기준 불명확 커밋 단위 분리 어려움
좋은 요청 UserService의 중복 validation 로직만 별도 함수로 분리해줘. 동작은 바꾸지 말고, 관련 테스트가 있으면 함께 업데이트해줘. 수정 파일은 service와 test로 제한 기존 API 시그니처 유지 diff에서 확인 가능한 변경만 적용
Git 관점좋은 AI 요청은 좋은 커밋 단위와 비슷함. 하나의 요청은 하나의 목적

선택지
의미
주의점
Accept Current
현재 작업 브랜치의 변경 선택
incoming 쪽에서 필요한 로직이 사라질 수 있음
Accept Incoming
병합하려는 쪽의 변경 선택
내 작업 중 필요한 변경이 사라질 수 있음
Accept Both
두 변경 모두 포함
중복 코드와 순서 문제 확인 필요
Compare Changes
양쪽 변경 비교
최종적으로 맞는 코드를 다시 구성
핵심충돌 해결은 버튼 선택이 아니라 최종 코드를 만드는 일. AI에게 맡겨도 최종 diff는 사람이 확인

Checklist

AI 시대 Git 체크리스트

전
AI 작업 전
Git 상태 정리, 필요한 변경 미리 커밋, 새 브랜치 시작, 요청 범위 축소
후
AI 작업 후
변경 파일 목록 확인, diff 직접 검토, 의도하지 않은 삭제와 설정 변경 확인, 테스트 실행
협
협업 전
의미 단위 stage, 구체적 커밋 메시지, PR 설명의 사실 확인, 비밀정보 포함 여부 확인

1
